# Blockmark Specification v0.1

## Abstract

Blockmark extends CommonMark with **block-level identifiers**: a lightweight prefix
syntax that gives any block a stable, addressable ID. The goal is to let tools —
especially AI agents — target, extract, and patch individual blocks without
full-document rewrites.

## 1. Syntax

### 1.1 Block ID Prefix

A **block ID prefix** is a token of the form:

```
^[<id>]
```

or with optional attributes:

```
^[<id> <key>=<value> ...]
```

It appears at the **very start** of a block-level construct (paragraph, heading,
list item, code fence, blockquote, thematic break) separated from the block
content by exactly one space.

### 1.2 Formal Grammar (EBNF)

```ebnf
blockmark_prefix  = "^[" , id , { " " , attribute } , "] " ;
id                = id_start , { id_continue } ;
id_start          = ALPHA | "_" ;
id_continue       = ALPHA | DIGIT | "-" | "_" | "." ;
attribute         = attr_name , "=" , attr_value ;
attr_name         = ALPHA , { ALPHA | DIGIT | "-" | "_" } ;
attr_value        = unquoted_value | quoted_value ;
unquoted_value    = ( ALPHA | DIGIT | "-" | "_" | "." | "/" | ":" )+ ;
quoted_value      = '"' , { any_char - '"' } , '"' ;
ALPHA             = "a"-"z" | "A"-"Z" ;
DIGIT             = "0"-"9" ;
```

### 1.3 Constraints

1. The `^[` must appear at column 0 of the line (or after list/blockquote
   markers in their respective contexts).
2. The closing `] ` (bracket + space) separates the prefix from content.
   The space is mandatory.
3. IDs must be unique within a document. Parsers SHOULD warn on duplicates.
4. IDs are case-sensitive: `^[Req-1]` and `^[req-1]` are distinct.
5. The prefix is metadata — it is **not** part of the block's rendered content.

### 1.4 Where Block IDs Can Appear

| Block type      | Example                                          |
|-----------------|--------------------------------------------------|
| Paragraph       | `^[intro] This is the introduction.`             |
| ATX heading     | `^[arch] ## Architecture`                        |
| List item       | `^[task-1] - [ ] Implement auth`                 |
| Fenced code     | `^[snippet-1] \`\`\`python`                     |
| Blockquote      | `^[note-1] > Important caveat here.`             |
| Thematic break  | `^[sep-1] ---`                                   |

Block IDs do **not** apply to:
- Inline content (emphasis, links, code spans)
- HTML blocks
- Link reference definitions
- Blank lines

## 2. Semantics

### 2.1 Identity

A block ID establishes a **stable handle** for the block it prefixes. The ID
persists across edits to the block's content — changing the text of `^[req-1]`
does not change its identity.

### 2.2 Attributes

Attributes are optional key-value metadata attached to the block. They provide
structured hooks for tooling without affecting rendering.

Reserved attribute names (semantic meaning defined by this spec):

| Attribute    | Meaning                                    |
|--------------|--------------------------------------------|
| `type`       | Classification (e.g., `requirement`, `decision`, `task`) |
| `status`     | Lifecycle state (e.g., `open`, `done`, `deprecated`)     |
| `supersedes` | ID of a block this one replaces            |
| `depends`    | Comma-separated IDs this block depends on  |

Tools MAY define additional attributes. Unknown attributes MUST be preserved
during round-tripping.

### 2.3 Scope

A block ID applies to the **entire block** it prefixes, including all
continuation lines. For a list item, it covers the item and any nested content.
For a heading, it covers only the heading line itself (not subsequent paragraphs).

## 3. Rendering

### 3.1 HTML Output

Block IDs are **stripped** from rendered output by default. The block renders
as if the prefix were not present. Renderers MAY optionally:

- Add an `id` attribute to the block's HTML element: `<p id="req-1">...</p>`
- Add a `data-blockmark-id` attribute for tooling
- Add `data-blockmark-*` attributes for each blockmark attribute

### 3.2 Plain Text / Markdown Output

When serializing back to markdown, the prefix MUST be preserved exactly as
written. Round-trip fidelity (parse then stringify producing byte-identical
output) is a hard requirement.

## 4. Parsing Rules

### 4.1 Recognition

The parser recognizes a block ID prefix when:

1. The line starts with `^[` (after optional block-level markers like `>` or `-`)
2. Followed by a valid ID
3. Optionally followed by space-separated `key=value` attributes
4. Closed by `] ` (bracket + exactly one space)
5. Followed by block content on the same line (or a code fence opener)

If any of these conditions fail, the `^[...]` is treated as literal text
(graceful degradation).

### 4.2 Ambiguity Resolution

- `^[not closed` — no closing bracket → literal text
- `^[id]no-space` — no space after bracket → literal text
- `^[id] ` followed by blank → valid empty-content block
- `^[]` — empty ID → literal text (IDs must be non-empty)
- `^[id key=]` — empty attribute value → literal text

### 4.3 Interaction with CommonMark

Block ID prefixes are parsed **before** standard CommonMark block structure.
The content after the prefix is parsed normally. This means:

```markdown
^[h1] # Hello World
```

produces a heading with ID `h1` and text "Hello World", not a paragraph.

## 5. Examples

### 5.1 Requirements Document

```markdown
# Product Requirements

^[req-1 type=requirement status=open] Users must authenticate via OAuth 2.0.

^[req-2 type=requirement status=done] The dashboard must load in under 2 seconds.

^[req-3 type=requirement status=open depends=req-1] Admin users can manage
other users' permissions.
```

### 5.2 Architecture Decision Record

```markdown
^[adr-title] # ADR-001: Database Selection

^[context] ## Context
We need a database that supports JSONB and full-text search.

^[decision type=decision supersedes=adr-000-decision] ## Decision
We chose PostgreSQL for its mature JSONB support.

^[status type=status] ## Status
Accepted — 2024-10-01.
```

### 5.3 Agent Interaction

```bash
# Read a specific block
blockmark get doc.md req-1
# → "Users must authenticate via OAuth 2.0."

# Patch a block surgically
blockmark patch doc.md req-1 --content "Users must authenticate via SSO."

# Query blocks by type
blockmark query doc.md --type=requirement --status=open
# → req-1, req-3

# Diff two versions by block
blockmark diff v1.md v2.md
# → req-1: changed, req-2: unchanged, req-3: added
```

## 6. Versioning

This is **v0.1** of the Blockmark specification. Future versions may add:

- Inline ID spans (for sub-block addressability)
- Block groups / sections
- Cross-document references (`doc.md#req-1`)
- Schema definitions for attribute validation

Changes to the syntax or semantics will follow semver: breaking changes
increment the major version.
